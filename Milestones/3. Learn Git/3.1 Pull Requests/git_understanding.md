# Git Understanding

## Why are PRs important in a team workflow?

they are important because if someone pushes unfinished code or the one that holds bugs, errors or incorrect output it might break the workflow for everyone. So instead of pushing work into the main branch devs work on seperate branches to make sure everything is reviewed and okay before merging. This way mistakes are caught up early on and code maintains its integrity. 

## What makes a well-structured PR?

- have a clear and meaningful title
- explain what was changed and why
- be focused on one small task at a timw
- include testing notes if relevant
- link to a related issue when applicable

## What did I learn from reviewing an open-source PR?

they are more detailed than small personal project PRs. Many of them had a clear summary, explanation of the problem, the solution, and a test plan. I also noticed that open-source projects often use automated checks, contributor license checks, and review comments before changes are accepted. Those PRs were used not only for merging code, but also for communication, collaboration, and maintaining code quality.
